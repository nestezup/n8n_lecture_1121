n8n의 3가지 핵심 개념

1. 워크플로우 (Workflow)

자동화의 전체 시나리오
하나의 완성된 자동화 프로세스

2. 노드 (Node)

각각의 작업 단위 (레고 블록처럼 조립)
데이터 입력, 변환, 출력 등 개별 기능 수행

3. 트리거 (Trigger)

자동화를 시작시키는 신호
시간, 이벤트, 웹훅 등 다양한 방식

    차이: difference,

    나눗셈: quotient.toFixed(2)
  }
};

Switch vs IF 비교

IF 노드

  • 용도: 2개 경로 분기 (예/아니오)
  • 조건: 단일 조건 평가
  • 예시: 승인/거부, 정상/오류, 주중/주말

Switch 노드

  • 용도: 3개 이상 경로 분기
  • 조건: 여러 조건을 순차적으로 평가
  • 예시: 등급 분류, 우선순위 분류, 상태별 처리

실전 활용:

  • 고객 유형별로 다른 메시지 전송
  • 주문 금액에 따른 배송비 계산
  • 업무 중요도에 따른 담당자 배정
📥 Switch 실전 예제:
1. 고객 유형별 메시지 | 2. 배송비 계산 | 3. 담당자 배정

API 이해하기

API (Application Programming Interface)

= 다른 서비스의 기능과 데이터를 사용할 수 있게 해주는 인터페이스

API의 역할:

  • 외부 서비스에 데이터 요청
  • 요청에 대한 응답 수신
  • 정해진 규칙(프로토콜)에 따라 통신

API 활용 예시:

  • 날씨 정보 조회
  • 환율 정보 확인
  • 지도에서 주소 검색
  • 텍스트 번역

OpenWeatherMap API 키 발급 방법

1. 회원가입

  • https://openweathermap.org/api 접속
  • 우측 상단 "Sign In" → "Create an Account" 클릭
  • 이메일, 비밀번호, 사용자명 입력 후 가입

2. API 키 확인

  • 로그인 후 "API keys" 메뉴 클릭
  • 기본 API 키가 자동 생성되어 있음 (또는 "Generate" 버튼 클릭)
  • API 키 복사 (활성화까지 최대 2시간 소요)

3. 무료 플랜

  • 분당 60회 호출 가능
  • 현재 날씨, 5일 예보 등 기본 기능 사용 가능
오늘 실습용 API 키: 강의에서 제공하는 공용 키를 사용합니다

OpenWeatherMap API 호출 예시

기본 호출 URL (One Call API 3.0)

https://api.openweathermap.org/data/3.0/onecall?lat=37.5665&lon=126.9780&exclude=minutely,hourly,daily,alerts&appid=YOUR_API_KEY&units=metric&lang=kr

파라미터 설명:

  • lat: 위도 (예: 서울 37.5665)
  • lon: 경도 (예: 서울 126.9780)
  • exclude: 제외할 데이터 (minutely, hourly, daily, alerts)
  • appid: 발급받은 API 키
  • units: 온도 단위 (metric=섭씨, imperial=화씨)
  • lang: 응답 언어 (kr=한국어, en=영어)

주요 도시 좌표:

  • 서울: lat=37.5665, lon=126.9780
  • 부산: lat=35.1796, lon=129.0756
  • 제주: lat=33.4996, lon=126.5312

표현식 {{ }} 이해하기

표현식의 구조

  • {{ $json.필드명 }}: 이전 노드의 데이터 참조
  • {{ $json.위도 }}: 단순 필드 접근
  • {{ $json.current.temp }}: 계층 구조 접근 (점 표기법)
  • {{ $json.current.weather[0].description }}: 배열 접근 (인덱스 사용)

실습 6 예시:

  • Merge 노드 후 데이터: {{ $json.위도 }}, {{ $json.경도 }}, {{ $json.API키 }}
  • API 응답 데이터: {{ $json.current.temp }}, {{ $json.current.humidity }}

주요 변수:

  • $json: 이전 노드의 JSON 출력 데이터
  • $input: 전체 입력 데이터 (여러 항목 포함 가능)
  • $node: 특정 노드의 데이터 참조

구글 시트 활용 방안

간단한 데이터베이스 역할

  • 소규모 데이터 저장 및 관리
  • 팀원들과 쉽게 공유 가능
  • 별도 데이터베이스 서버 불필요

실전 조합 패턴:

  • 웹훅으로 데이터 수신 → 시트에 자동 저장
  • 폼 제출 내용 → 시트에 기록
  • 정기 리포트 생성 → 시트에 작성

주의사항:

  • 대용량 데이터(수만 행 이상)는 실제 데이터베이스 권장
  • 동시 쓰기 작업이 많으면 충돌 가능
  • 민감한 데이터는 보안 고려 필요

자동화 가능성 판단

자동화하기 적합한 작업

  • 규칙이 명확하고 일관됨
  • 자주 반복됨 (일, 주, 월 단위)
  • 데이터 기반 작업
  • 여러 단계로 구성

자동화가 어려운 작업

  • 매번 다른 판단이 필요
  • 창의성이나 감성이 중요
  • 규칙이 모호하거나 예외가 많음
  • 한 번만 수행하는 작업

판단 기준:

  • 시간 절약 효과
  • 에러 감소 효과
  • 구현 난이도

자동화 시작 전략

1단계: 작게 시작

  • 2-3개 노드로 구성 가능한 것부터
  • 한 번에 하나씩만 자동화
  • 성공 경험을 쌓기

2단계: 점진적 확장

  • 기본 동작이 확인되면 기능 추가
  • 조건 분기 추가
  • 다른 앱 연결

3단계: 최적화

  • 에러 처리 추가
  • 알림 방식 개선
  • 성능 향상

완벽을 추구하지 말고 작동하는 것부터 만들어보세요.

디버깅 방법

데이터 흐름 확인:

  • 각 노드를 클릭하여 Input/Output 탭 확인
  • 문제가 있는 노드 이전에 Set 노드를 추가하여 데이터 구조 파악
  • 표현식이 올바른 값을 참조하는지 확인

단계적 테스트:

  • Manual Trigger로 수동 실행하며 단계별 확인
  • 작은 데이터셋으로 먼저 테스트
  • 한 번에 하나씩 노드 추가하며 검증

에러 메시지 활용:

  • 에러 메시지를 자세히 읽기
  • 구글 검색: "n8n [에러 메시지]"
  • n8n 커뮤니티 포럼에서 검색
  • ChatGPT나 Claude에게 에러 메시지 설명 요청

Webhook

변형/정리
